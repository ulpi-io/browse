## Overview

Fix 4 confirmed bugs: screenshot-diff broken in Bun runtime, domain filter route.continue() blocks user routes, --allowed-domains only injects into active tab, and passwords starting with `--` rejected. All bugs have been verified against live code.

## Scope Challenge

The screenshot-diff fix is the most complex. Playwright's internal modules (`playwright-core/lib/server/utils/comparators`) don't resolve in Bun's `require()` system, and absolute path resolution fails in compiled binaries (`$bunfs`). The proper fix is a self-contained PNG decoder using `zlib.inflateSync()` — verified to produce byte-identical output to Playwright's decoder. The other 3 fixes are 1-5 line changes each.

## Tasks

### TASK-001: Create src/png-compare.ts — self-contained PNG decoder + pixel comparator

Create a new file `src/png-compare.ts` that decodes PNG files to raw RGBA pixels and compares them pixel-by-pixel. No external dependencies — uses only `zlib.inflateSync()` from Node/Bun standard library. Must work in both `bun run` dev mode and compiled binary.

**Implementation:**
- `decodePNG(buf: Buffer): { width, height, data: Buffer }` — parse IHDR, collect IDAT chunks, inflate, unfilter scanlines (all 5 filter types: None/Sub/Up/Average/Paeth), output RGBA
- `compareScreenshots(baseline: Buffer, current: Buffer, threshold?: number): { totalPixels, diffPixels, mismatchPct, passed, diffImage?: Buffer }` — decode both, compare pixel-by-pixel using Euclidean RGB distance
- Support color types 2 (RGB) and 6 (RGBA)
- Validate PNG magic bytes, reject non-8-bit or unsupported color types with clear error
- Generate simple diff image: copy current image, tint differing pixels red

Verified: the decoder produces byte-identical output to Playwright's `PNG.sync.read()` on a 1920x1080 screenshot.

**Type:** feature
**Effort:** M

**Acceptance Criteria:**
- [ ] `decodePNG()` correctly decodes a Playwright-generated PNG (all 5 filter types) to RGBA buffer
- [ ] `compareScreenshots()` returns `passed: true` when comparing identical images
- [ ] Comparing two different screenshots returns `passed: false` with correct mismatch percentage
- [ ] Throws clear error on non-PNG input or unsupported color type (e.g., palette-based)

**Agent:** nodejs-cli-senior-engineer

**Priority:** P0

### TASK-003: Fix domain filter to use route.fallback() instead of route.continue()

Replace every `route.continue()` in domain filter handlers with `route.fallback()`. This is the critical fix: `continue()` sends allowed requests directly to the network, bypassing user-registered block/fulfill handlers. `fallback()` passes control to the next matching handler in the chain.

**Files to modify:**
- `src/commands/write.ts` — `rebuildRoutes()` function (line 34) and inline handler (line 447)
- `src/browser-manager.ts` — `recreateContext()` domain filter re-registration (line 593)
- `src/session-manager.ts` — initial domain filter setup (line 56) and existing session update (line 97)

Change every instance of:
```typescript
if (domainFilter.isAllowed(url)) { route.continue(); }
```
to:
```typescript
if (domainFilter.isAllowed(url)) { route.fallback(); }
```

**Type:** bug
**Effort:** S

**Acceptance Criteria:**
- [ ] User `route block` handler fires for allowed-domain URLs when `--allowed-domains` is active
- [ ] User `route fulfill` handler returns mocked response for allowed-domain URLs
- [ ] Blocked-domain URLs are still aborted (no regression)

**Agent:** nodejs-cli-senior-engineer

**Priority:** P1

### TASK-004: Inject domain filter initScript into all open tabs, not just active

In `src/session-manager.ts`, when adding `--allowed-domains` to an existing session, iterate all open tabs and inject the init script into each one via `page.evaluate()`, not just the active page.

**Files to modify:** `src/session-manager.ts`

Replace:
```typescript
const page = session.manager.getPage();
await page.evaluate(initScript);
```
with:
```typescript
const tabs = session.manager.getTabList();
for (const tab of tabs) {
  try {
    const page = session.manager.getPageById(tab.id);
    if (page) await page.evaluate(initScript);
  } catch {}
}
```

Need to add `getPageById(id: number): Page | undefined` to BrowserManager if it doesn't exist.

**Files to modify:** `src/session-manager.ts`, `src/browser-manager.ts`

**Type:** bug

### TASK-005: Accept passwords starting with -- in auth save

In `src/commands/meta.ts` auth save handler, the flag scanner currently drops positional args starting with `--`. Fix: use `--` as a separator (standard POSIX convention). Everything after a bare `--` arg is treated as positional, not flags. If no `--` separator, treat the first positional after username as the password regardless of prefix.

**Files to modify:** `src/commands/meta.ts`

Change the positional collection logic: after scanning known flags (`--user-sel`, `--pass-sel`, `--submit-sel`), treat any remaining arg as positional, even if it starts with `--`.
