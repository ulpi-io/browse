# Code of Conduct

## Our Pledge

We are committed to making participation in this project a welcoming experience for everyone, regardless of age, body size, disability, ethnicity, gender identity and expression, level of experience, nationality, personal appearance, race, religion, or sexual identity and orientation.

## Our Standards

### Expected behavior

- **Be respectful.** Disagreement is fine; disrespect is not. Critique ideas, not people.
- **Be constructive.** When reviewing code or responding to issues, explain *why* and suggest alternatives. "This is wrong" is not helpful; "This could break session isolation because X — consider Y instead" is.
- **Be welcoming.** Remember that everyone was new once. Answer questions patiently. Point people to docs before assuming they should already know.
- **Give and accept feedback gracefully.** Your PR might get pushback. That's the process working, not a personal attack. Similarly, when reviewing, assume the author made their choices for a reason — ask before assuming.
- **Focus on the project.** Keep discussions relevant to the work. Political, religious, and other off-topic debates belong elsewhere.
- **Credit others.** If your contribution builds on someone else's idea or code, acknowledge it.

### Unacceptable behavior

- Personal attacks, insults, or derogatory comments — in issues, PRs, or any project channel
- Trolling, baiting, or deliberately inflammatory remarks
- Sustained disruption of discussions (e.g., repeatedly derailing issue threads)
- Publishing others' private information (addresses, emails, etc.) without explicit permission
- Unwelcome sexual attention or advances
- Advocating for or encouraging any of the above
- Any other conduct that would be considered inappropriate in a professional workplace

## Scope

This Code of Conduct applies to all project spaces:

- GitHub issues, pull requests, and discussions
- Code review comments
- Commit messages (yes, really)
- Any other forum where you represent this project (social media, conferences, chat)

It also applies when an individual is officially representing the project in public spaces.

## Enforcement

### Process

1. **Report** — Contact the project maintainers via [GitHub Security Advisories](https://github.com/ulpi-io/browse/security/advisories/new) (for private reports) or by emailing the maintainer listed in the project's GitHub profile. All reports are confidential.
2. **Review** — Maintainers will review the report, gather context, and determine what happened. The reporter will not be identified to the accused without consent.
3. **Action** — Based on severity and context, maintainers may take any of the following actions:

| Action | When |
|--------|------|
| Private warning | First minor offense, clearly unintentional |
| Public warning | Pattern of minor offenses, or single offense visible to the community |
| Temporary ban | Serious offense or repeated violations after warnings |
| Permanent ban | Severe offense, or continued violations after temporary ban |

### Maintainer responsibilities

Maintainers are responsible for clarifying and enforcing these standards. They have the right and responsibility to remove, edit, or reject comments, commits, code, wiki edits, issues, and other contributions that violate this Code of Conduct.

Maintainers who do not follow or enforce the Code of Conduct in good faith may face temporary or permanent repercussions as determined by other project leaders.
